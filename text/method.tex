\section{Methodology}
\label{sec:method}

\subsection{Workload Taxonomy}
\label{sec:taxonomy}

We organize inference workloads into three tiers (Table~\ref{tab:profiles}):

\begin{itemize}
    \item \textbf{Tier~1: Agentic traces.} Real request logs from SWE-Bench~\citep{swebench} coding agents and TerminalBench CLI agents, in both single-turn (ISL$\approx$17K, OSL$\approx$800, ratio 21:1) and multi-turn variants (13--876 steps, growing context up to 131K tokens, ratio 33:1). These capture the high ISL/OSL ratios and context accumulation that define agentic workloads.

    \item \textbf{Tier~2: Chat baselines.} ShareGPT-derived conversations at three scales: short (ISL=500, OSL=300), medium (ISL=2{,}000, OSL=1{,}000), and long (ISL=8{,}000, OSL=2{,}000). These represent current benchmark practice and serve as our control group.

    \item \textbf{Tier~3: Synthetic stress tests.} Controlled workloads isolating specific bottlenecks: decode-heavy (ISL=256, OSL=4{,}096) for memory-bandwidth stress, prefill-heavy (ISL=8{,}192, OSL=256) for compute stress, and balanced (ISL=1{,}024, OSL=1{,}024).
\end{itemize}

\subsection{Benchmarking Framework}
\label{sec:framework}

We build on an open-source asynchronous benchmarking tool that sends concurrent requests to OpenAI-compatible serving endpoints. For each (model, profile, concurrency) configuration, we execute 50 requests after a 3-request warmup and record per-request metrics: time-to-first-token (TTFT), inter-token latency (ITL), time-per-output-token (TPOT), end-to-end latency, and input/output token counts. We report median, p90, and p99 percentiles for latency metrics and aggregate throughput in output tokens per second.

Concurrency is swept from 1 (single request, no batching) through 10, 40, 120, and 320 concurrent requests to construct saturation curves. At concurrency~1, metrics reflect single-request latency without scheduling interference; at high concurrency, metrics reflect the system's throughput ceiling and queueing behavior.

\subsection{Kernel-Level Profiling}
\label{sec:ncu_method}

To explain \emph{why} different workload profiles produce different throughput characteristics, we profile the CUDA kernels executed during model forward passes using NVIDIA Nsight Compute (ncu, v2024.3.2) with the \texttt{--set full} flag, which collects 47 replay passes per kernel to measure:
\begin{itemize}
    \item \textbf{Compute throughput}: SM active cycles, tensor core utilization, achieved TFLOPS
    \item \textbf{Memory throughput}: DRAM bytes read/written, L2 cache hit rate, achieved HBM bandwidth
    \item \textbf{Kernel duration}: wall-clock execution time in microseconds
\end{itemize}

We profile two complementary datasets: (1)~56 isolated GEMM shapes at batch dimensions $M \in \{1, 2, 4, \ldots, 8192\}$ with weight dimensions matching LLM architectures ($N,K \in \{4096, 8192, 14336\}$), providing a systematic roofline sweep; and (2)~full 2-layer forward passes of Llama-3.1-8B (419 kernels) and Mixtral-8$\times$7B (1,527 kernels) at sequence length 512, capturing the complete kernel mix including flash attention, RMSNorm, SiLU activation, residual additions, and MoE expert routing.

From the ncu data, we compute each kernel's \emph{arithmetic intensity} (AI = FLOPs / bytes transferred) and plot it against achieved throughput on the A100 roofline, classifying kernels as compute-bound (AI $> 153$, the A100's ridge point at 312 TFLOPS / 2039 GB/s) or memory-bound (AI $< 153$).

\subsection{Experimental Setup}
\label{sec:setup}

\paragraph{Hardware.}
We evaluate on two GPU platforms: (1)~8$\times$ NVIDIA A100-SXM4-40GB with NVLink interconnect (on-premise), and (2)~2$\times$ NVIDIA H100-SXM5-80GB with NVLink (cloud). The A100 provides 312 TFLOPS (bf16) and 2,039 GB/s HBM2e bandwidth; the H100 provides 989 TFLOPS (bf16) and 3,352 GB/s HBM3 bandwidth.

\paragraph{Serving engines.}
We use vLLM~v0.19 as the primary serving engine (continuous batching, paged attention, CUDA graphs for decode) with SGLang as a secondary comparison on H100. All models are served in bfloat16 precision.

\paragraph{Models.}
Table~\ref{tab:models} summarizes the 9 model configurations evaluated, spanning dense transformers (Llama-3.1-8B/70B, Qwen-2.5-72B) and mixture-of-experts architectures (Mixtral-8$\times$7B, gpt-oss-20b with 32 experts, gpt-oss-120b with 128 experts). Tensor parallelism ranges from TP=1 to TP=4 depending on model size and available memory.
