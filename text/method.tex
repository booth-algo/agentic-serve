\section{Methodology}
\label{sec:method}

We benchmark 6 LLMs spanning 8B--120B parameters across dense (Llama-3.1-8B/70B, Qwen-2.5-72B) and mixture-of-expert (Mixtral-8$\times$7B, gpt-oss-20b, gpt-oss-120b) architectures. Inference is served on H100-SXM5-80GB and A100-SXM4-40GB GPUs (plus RTX~3090 for the small models) using vLLM v0.19.0 and SGLang in bfloat16 precision. Concurrency sweeps 1--320 simultaneous requests across 20 workload profiles, with $N{\geq}50$ requests after 3 warmup requests per configuration. Section~\ref{sec:taxonomy} defines the workload profiles, \ref{sec:framework} the benchmarking framework, \ref{sec:ncu_method} the kernel-level profiling, \ref{sec:perop_method} the per-operator and per-kernel predictors, and \ref{sec:setup} the full experimental setup.

\subsection{Workload Taxonomy}
\label{sec:taxonomy}

Workloads fall into three tiers (Table~\ref{tab:profiles}).

\paragraph{Tier 1: Agentic traces.} Real request logs from SWE-Bench~\citep{swebench} coding agents (165 sessions, 13--320 turns, median 85), TerminalBench CLI agents (267 sessions, 2--876 turns, median 61), and OSWorld computer-use agents (60 sessions). Multi-turn variants are bucketed by turn count (\texttt{short}/\texttt{medium}/\texttt{long}/\texttt{xl}) and capped at 32K, 65K, or 131K tokens of accumulated context. A single-turn \texttt{coding-agent} profile (ISL$\approx$17K, OSL$\approx$800) acts as the agentic baseline.

\paragraph{Tier 2: Chat baselines.} ShareGPT conversations at three single-turn scales (ISL 500/2K/8K, OSL 300/1K/2K) and three multi-turn scales (3--20 turns, max context 8K--32K). Source: \texttt{Aeala/ShareGPT\_Vicuna\_unfiltered}, filtered by an approximate token count (word\_count$\times$1.35); the listed ISL is an upper-bound filter, not an exact per-request length.

\paragraph{Tier 3: Synthetic stress tests.} Random-token workloads isolating specific bottlenecks: \texttt{decode-heavy} (ISL=256, OSL=4{,}096) for memory-bandwidth stress, \texttt{prefill-heavy} (ISL=8{,}192, OSL=256) for compute stress, and \texttt{random-1k} (ISL=1{,}024, OSL=1{,}024) as a balanced reference.

\paragraph{Single-turn vs.\ multi-turn semantics.} For single-turn profiles, ISL is the (approximate) per-request input length. For multi-turn profiles, ISL is the \emph{maximum context window cap}: turn~1 starts with system + first user message ($\sim$2K tokens), and context grows by user/assistant pairs each turn until the cap or session end.

\subsection{Benchmarking Framework}
\label{sec:framework}

We extend an open-source asynchronous benchmark client that targets OpenAI-compatible endpoints (\texttt{/v1/chat/completions}). For each (model, profile, concurrency) cell, we record per-request TTFT, ITL, TPOT, end-to-end latency, and input/output token counts from the streamed response. We report median and p90; throughput is aggregate output tokens per second. Three warmup requests per cell are discarded.

\paragraph{Closed-loop arrival.} Concurrency is enforced via \texttt{asyncio.Semaphore(concurrency)}: all $N$ requests fire at $t{=}0$ but at most \texttt{concurrency} are in flight, with the next request launched the instant one completes. Open-loop \texttt{--request-rate inf} runs (used by InferenceX) inflate TTFT with queue-wait time; our TTFT reflects scheduler behavior at a stable in-flight load.

\paragraph{Concurrency and request budget.} We sweep $\{1, 10, 20, 40, 80, 120, 160, 200, 256, 320\}$ on A100/H100 and $\{1, 10, 20, 40, 80\}$ on RTX~3090. Request count scales with concurrency: $N{=}50$ at conc.\ 1, $N{=}200$ at conc.\ 5--80, $N{=}\max(200, 2\cdot\text{conc.})$ at conc.\ ${\geq}120$. This keeps the system at steady state across operating points.

\paragraph{Statistical caveat.} Each cell runs once with the $N$ requests above. Percentiles are over those requests, not multiple repeats. We omit p99: at $N{=}50$ it reduces to a single sample.

\subsection{Kernel-Level Profiling}
\label{sec:ncu_method}

To explain \emph{why} workloads diverge, we profile the CUDA kernels of model forward passes with NVIDIA Nsight Compute (ncu, v2024.3.2, \texttt{--set full}). Each kernel reports compute throughput (SM active cycles, tensor-core utilization, achieved TFLOPS), memory throughput (DRAM bytes, L2 hit rate, achieved HBM bandwidth), and wall-clock duration.

We collect two complementary datasets. \textbf{2-layer model passes}: Llama-3.1-8B (417 kernels), Mixtral-8$\times$7B (1{,}525), and two unseen held-out models, Qwen-2.5-72B and Llama-3.3-70B (421 each). \textbf{GEMM shape sweeps}: 56 isolated GEMMs at $M \in \{1,\ldots,8192\}$, $N,K \in \{4096,8192,14336\}$ (372 kernels), plus a wider hidden-size sweep covering $\{2048,\ldots,28672\}$ (451 kernels). Total: 2{,}811 profiled kernels, 842 held out for validation. All profiles run at batch~1, sequence length 128.

\paragraph{Why 2 layers suffice.} The kernel mix is identical across transformer blocks: same operators, same shapes. Only KV-cache size grows with depth, but our prefill-only profiles carry no accumulated cache. Summing 2-layer kernel times and scaling by $n_\text{layers}/2$ matches a full forward pass within $+0.1$--$0.2\%$.

\paragraph{Roofline classification.} For each kernel we compute arithmetic intensity (AI = FLOPs / bytes) and plot it against achieved throughput. The A100 dense bf16 ridge sits at 312~TFLOPS / 2{,}039~GB/s $=$ \textbf{153 FLOPs/byte}; AI $> 153$ is compute-bound, AI $< 153$ memory-bound. At our profiling configuration (bs=1, seq=128) every kernel is memory-bound. The AI distribution shifts right as $M$ grows under continuous batching --- the regime studied in Section~\ref{sec:roofline}.

\subsection{Per-Operator and Per-Kernel Latency Predictors}
\label{sec:perop_method}

To estimate latency without a fresh GPU run for every new model or shape, we train two XGBoost regressors at complementary granularities.

\paragraph{Level 1 -- per-kernel predictor.} Trained on 2{,}811 ncu-profiled kernels with a log-transformed latency target. The 41 features (Table~\ref{tab:features_l1}) combine raw ncu counters with derived quantities and a one-hot kernel-family encoding. Hyperparameters: \texttt{n\_estimators=500}, \texttt{max\_depth=6}, \texttt{lr=0.03}.

\subsection{Experimental Setup}
\label{sec:setup}

\paragraph{Hardware.} 8$\times$ NVIDIA A100-SXM4-40GB on-premise (NVSwitch, \texttt{nvidia-fabricmanager}); 2--4$\times$ NVIDIA H100-SXM5-80GB on RunPod (NVLink); 8$\times$ NVIDIA RTX~3090 24GB on-premise (PCIe, no NVLink). A100: 312 TFLOPS bf16, 2{,}039 GB/s HBM2e. H100: 989 TFLOPS bf16, 3{,}352 GB/s HBM3 (1.6$\times$ bandwidth, 3.2$\times$ compute over A100). Each run uses the GPU subset required by its TP degree and runs in isolation.

\paragraph{Software.} vLLM v0.19.0; SGLang as a secondary comparison on H100; CUDA 12.6 (A100, ncu) / 13.0 (RTX 3090); NVIDIA driver 580.126.09; PyTorch 2.10; XGBoost 3.2.0; ncu v2024.3.2; Python 3.12.

\paragraph{Serving flags.} On A100 we launch vLLM with \texttt{--enforce-eager --disable-custom-all-reduce --no-enable-log-requests --dtype bfloat16 --gpu-memory-utilization 0.95}; CUDA-graph capture OOMs at our memory budget, and the custom all-reduce kernel is unstable on A100-SXM4. \texttt{--max-model-len} is 4096 for gpt-oss-20b, 2048 for the larger models. On H100 we use vLLM defaults (CUDA graphs on, \texttt{--dtype bfloat16}). Prefix caching is off; chunked prefill is on by default in vLLM 0.19. All bf16 except gpt-oss-20b/120b, which use the released MXFP4 weights.

\paragraph{Tokenization.} Requests go over the OpenAI-compatible chat API; the engine handles tokenization and chat templating with each model's native tokenizer. ShareGPT and agent traces are filtered to an upper bound of word\_count$\times$1.35 tokens, so the ISL in Table~\ref{tab:profiles} is a filter cap, not an exact per-request count. Synthetic profiles sample tokens uniformly from the vocabulary.

\paragraph{Models.} Table~\ref{tab:models} summarizes the 6 architectures evaluated; Table~\ref{tab:tp_configs} lists the TP degrees actually run on each platform.
